# 寫作規範 (Author Guidelines)

為了維持《日常與職場的演算法應用》一書的品質與閱讀體驗一致性，請在撰寫新章節或修改現有內容時，嚴格遵守以下規範。

## 一、 章節結構標準

每一章 (`book/chapters/chXX/index.md`) 都必須具備以下固定結構，以引導讀者循序漸進：

1. **情境引入**：描述一個生活中或工作上的真實痛點（例如：如何排定優先順序、如何處理重疊的會議室），讓讀者產生共鳴。
2. **問題定義 (輸入/輸出/限制)**：將痛點抽象化。明確定義我們擁有哪些資料（輸入）、我們的目標是什麼（輸出），以及有什麼資源限制。
3. **演算法思維/資料結構**：用白話文解釋解決此問題的核心觀念（如優先佇列、滑動視窗）。多使用表格或條列式步驟說明，避免過度學術的數學證明。
4. **實作與範例**：展示具體的 Python 程式碼，並搭配清晰的註解。
5. **執行說明**：必須在程式碼區塊前後附上「請在專案根目錄執行」的提示，確保讀者能順利重現結果。

## 二、 程式碼與測試管理

- **分離原則**：核心商業邏輯**不可**只寫在 Markdown 裡，必須存放在 `examples/chXX/` 目錄下成為獨立的 Python 模組。
- **可測試性**：每個 `examples/` 裡的邏輯，都必須在 `tests/` 目錄下有對應的 `test_chXX_*.py` 測試檔案，並確保 `pytest` 能夠全數通過。
- **展示原則**：在 Markdown 檔案展示程式碼時，應使用標準的 ` ```python ` 區塊，並盡量保持精簡、聚焦於核心邏輯。

## 三、 字數與品質控管

- **字數下限要求**：每個章節的正文內容（排除 ` ```python ` 程式碼區塊後）**建議不得低於 2000 字**。這是為了確保核心論述與演算法解析具有足夠的深度。發布檢查腳本將會自動計算字數，若低於此標準將會拋出警告。

## 四、 排版與語氣

- **語氣**：保持對話式、實用導向。將讀者視為需要解決日常工作問題的一般人，而非資工系學生。
- **MyST 語法應用**：善用 Jupyter Book 支援的區塊來標示重點：
  - 使用 ````{note}` 區塊來補充背景知識。
  - 使用 ````{warning}` 區塊來提醒常見錯誤或避坑指南。
